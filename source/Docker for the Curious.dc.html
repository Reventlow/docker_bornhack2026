## concepts

Why Containers Exist

“But it works
on my machine…”
— every developer, right before a deadline
>

An app is never just code. It drags along a runtime, libraries, config files, and quirks of the machine it grew up on.

>

Move it somewhere else and something is missing, or the wrong version, or in the wrong place.

>

A container packs the app together with everything it needs — one sealed unit that runs the same on any machine.

## concepts

What a Container Is

A normal process on your machine — running inside its own sealed little world.

isolated

Its own filesystem, network, and process list. It can’t see the rest of your machine.

lightweight

It shares your kernel — no fake hardware, no second OS. Starts in milliseconds.

portable

The same image runs identically on any machine with Docker — laptop, server, cloud.

# not magic: linux namespaces + cgroups, wearing a friendly interface

## concepts

Image vs. Container

image — the blueprint
read-only template
built in layers, shared and cached
versioned with tags — nginx:1.27
never changes while containers run
container — the running copy
a live instance of an image
adds a thin writable layer on top
start, stop, delete freely
run many from the same image

container: camp

## the mental model

Ephemeral by Design

immutable images

Running containers never change the image. Want the image different? Rebuild it.

disposable containers

Cattle, not pets. Delete and recreate without fear — a fresh one is always one command away.

data lives outside

Anything worth keeping stays on your machine and gets mounted in. That’s a volume — next task.

TASK 4 hands-on

The Same Thing, as a File

camp/compose.yaml
services:
  web:
    image: elohite/bornhack2026
    ports:
      - "8080:80"
you@laptop:~ 
$ mkdir camp && cd camp
$ nano compose.yaml
$ docker compose up -d
$ docker compose down

That run command was getting long. Compose writes it down instead.

>Readable, shareable, lives in git next to your code.
>up -d starts everything in the file. down stops and cleans it all up.
>The file is the documentation of how this thing runs.
# on older machines the command is docker-compose (with a hyphen)

TASK 5 hands-on

Mount a Volume

camp/compose.yaml
services:
  web:
    image: elohite/bornhack2026
    ports:
      - "8080:80"
    volumes:
      - ./html:/usr/share/nginx/html
you@laptop:~/camp 
$ mkdir html
$ nano html/index.html   # write anything
$ docker compose up -d
$ docker compose down && docker compose up -d
# refresh — your page survived
the colon rule
./html : /usr/share/nginx/html
your machine (host)inside the container

Left of the colon: your world. Right of it: theirs. Same rule you already used with ports — 8080:80.

# the container can be deleted a hundred times. ./html is yours and stays.

Two Services, One File

wordpress/compose.yaml
services:
  wordpress:
    image: wordpress
    ports:
      - "8081:80"
    environment:
      WORDPRESS_DB_HOST: db
      WORDPRESS_DB_USER: wp
      WORDPRESS_DB_PASSWORD: secret
      WORDPRESS_DB_NAME: wordpress
  db:
    image: mysql
    environment:
      MYSQL_DATABASE: wordpress
      MYSQL_USER: wp
      MYSQL_PASSWORD: secret
      MYSQL_RANDOM_ROOT_PASSWORD: "1"
>

Two containers, one file, one docker compose up -d.

>

Compose gives them a private network. Each service is reachable by name — WordPress finds its database at db.

>

A whole CMS with a database at localhost:8081 — nothing installed on the machine.

# in real life you’d give db a volume — you know why now
